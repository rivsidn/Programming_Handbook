
• 可以。先给一句最核心的话：

  作用域（scope）就是：一个变量名在什么地方能被访问到。

  在 Elisp 里，理解作用域要分两层看：

  - 这个变量是全局的还是局部的
  - 局部变量是按词法作用域还是动态作用域来查找

  ## 1. 全局变量和局部变量

  ### 全局变量

  在顶层定义或赋值的变量，通常可以在很多地方访问：

  (setq x 10)

  这里通常可以把 x 看成全局变量。

  更规范一点常用：

  (defvar x 10)

  ### 局部变量

  函数参数和 let 引入的变量，都是局部变量：

  (defun add (a b)
    (let ((c 100))
      (+ a b c)))

  这里：

  - a、b 是函数参数形成的局部变量
  - c 是 let 创建的局部变量

  它们只在对应代码块里有效。

  ———

  ## 2. 同名变量会发生“遮蔽”

  局部变量会暂时遮住外层同名变量：

  (setq x 1)

  (let ((x 2))
    x)
  ;; => 2

  x
  ;; => 1

  也就是说：

  - 在 let 里面看到的是局部 x
  - 离开 let 后，还是原来的全局 x

  ———

  ## 3. Elisp 最关键的区别：词法作用域 vs 动态作用域

  这是 Elisp 作用域里最重要的点。

  ### 词法作用域 lexical scope

  按代码的书写结构来决定变量可见性。

  现代 Elisp 一般推荐开启它，在文件第一行写：

  ;;; -*- lexical-binding: t; -*-

  开启后，let 和函数参数默认是词法变量。

  例子：

  ;;; -*- lexical-binding: t; -*-

  (setq x 1)

  (defun show-x ()
    x)

  (let ((x 2))
    (show-x))
  ;; => 1

  为什么结果是 1？

  因为 show-x 这个函数定义时，并没有被 let ((x 2)) ... 包住。
  所以它看不到调用处临时创建的这个局部 x，它用的是外面的全局 x。

  词法作用域的直觉就是：

  变量能不能看见，取决于它写在代码的哪一层里面。

  ———

  ### 动态作用域 dynamic scope

  按运行时调用链来找变量。

  老式 Elisp 或没有开启 lexical-binding 的代码里，局部绑定常常表现为动态作用域。

  同样的例子，在动态作用域下：

  (setq x 1)

  (defun show-x ()
    x)

  (let ((x 2))
    (show-x))
  ;; => 2

  为什么这里会变成 2？

  动态的，实时展开.

  因为在调用 show-x 的那一刻，外层正好有一个活动中的 x = 2，
  动态作用域会顺着当前调用过程去找这个绑定。

  动态作用域的直觉就是：

  不是看代码写在哪，而是看调用函数时，外面有没有一个当前生效的同名变量。

  ———

  ## 4. 为什么现代 Elisp 更推荐词法作用域

  因为它更符合大多数现代语言的直觉，也更容易推理。

  最重要的一点是：它支持闭包 closure。

  例如：

  ;;; -*- lexical-binding: t; -*-

  (let ((x 10))
    (setq f (lambda () x)))

  (funcall f)
  ;; => 10

  这里 lambda 记住了当时的 x，这就是闭包。

  如果没有词法作用域，这种“记住外层局部变量”的行为就不自然，很多情况下也不能这样工作。

  ———

  ## 5. 一个容易忽略的点：defvar 定义的是“特殊变量”

  即使你开启了 lexical-binding: t，用 defvar 声明的变量通常仍然是动态绑定的 special variable。

  例如：

  ;;; -*- lexical-binding: t; -*-

  (defvar my-level 1)

  (defun show-level ()
    my-level)

  (let ((my-level 2))
    (show-level))
  ;; 往往 => 2

  这是因为 my-level 被声明成了 special variable，它仍然按动态方式工作。

  所以要记住：

  - 普通 let 变量：在 lexical-binding: t 下通常是词法作用域
  - defvar / defcustom 这类声明出来的 special variable：通常仍是动态绑定

  ———

  ## 6. 你可以先这样记

  最实用的版本是这四句：

  - let 和函数参数会创建局部变量
  - 局部变量会遮蔽同名外层变量
  - 现代 Elisp 推荐 lexical-binding: t
  - defvar 声明的变量常常仍然是动态变量

  ———

  ## 7. 一个最小总结

  如果你只想抓住核心：

  Elisp 的作用域，就是变量名在哪儿可见。
  现代写法里：

  - 平常写的局部变量，多数按词法作用域
  - defvar 声明的变量，很多时候按动态作用域











## setq


› 回到变量，setq 设置的是全局变量么？


• 不一定。setq 的本质是：给变量赋值，它改的是“当前可见的那个绑定”。

  可以分成两种情况理解：

  1. 在顶层使用
     这时通常就是在设置全局值：

  (setq x 10)

  这里一般可以理解为：x 的全局值变成了 10。

  2. 在局部作用域里使用
     如果这个变量已经被 let 绑定成局部变量，那么 setq 改的是局部变量，不是全局变量：

  (setq x 1)   ;; 全局 x = 1

  (let ((x 2)) ;; 局部 x = 2
    (setq x 3) ;; 改的是局部 x
    x)
  ;; => 3

  x
  ;; => 1

  所以更准确地说：

  setq 不是“专门设置全局变量”的命令，而是“对变量赋值”的命令。

  - 没有局部绑定时，它通常改全局值
  - 有局部绑定时，它改局部值

  如果你想“声明一个全局变量”，更规范的写法通常是用 defvar：

  (defvar my-name "Tom"
    "示例全局变量。")

  然后再用 setq 改它：

  (setq my-name "Alice")

  你可以先记这句最实用的：

  顶层的 setq，通常就是在设全局变量；let 里面的 setq，通常是在改局部变量。
